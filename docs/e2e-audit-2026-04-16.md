**Status:** Decisions resolved, follow-up issues to create

## 1. Installation E2E (New Machine)

### Golden path (6 steps)

1. `uv tool install --from git+https://github.com/shawnoster/aya aya-ai-assist`
2. `aya init --label <name>`
3. `aya schedule install` (bundles: crontab + SessionStart/PreToolUse/PostToolUse hooks)
4. `aya pair --peer <name>` (optional, for relay)
5. Shell alias: `alias claude='claude --plugin-dir /path/to/aya'`
6. `aya status` (verify)

### Silent failures found

| Scenario | Location | Behavior | Severity |
|----------|----------|----------|----------|
| WSL without cron service | `install.py:257-267` | `crontab -l` fails silently; user gets misleading "Could not find 'aya' on PATH" | HIGH |
| `gh` CLI missing | `scheduler/providers.py:41-56` | Returns `None` silently; GitHub PR checks disappear with no warning | MEDIUM |
| `settings.json` missing | `install.py:380-381` | Returns `{}`; creates hooks anyway; recovers on save | LOW |
| coincurve build fails | `pyproject.toml:11` | Loud failure during `uv tool install`, but no pre-flight check or workaround docs | MEDIUM |

### Discoverability gaps

- `aya init` output has no next-steps guidance. Compare to `aya pair` which prints
  the pairing code and suggests the command for the other machine.
- `/aya-setup` skill exists in `.claude/commands/` but is **not mentioned** by
  `aya init`, `aya status`, or the plugin manifest. Users won't know it exists.
- Plugin alias (`--plugin-dir`) is documented in the setup skill but never validated.
  `aya status` doesn't check if the plugin is wired.

### WSL issues

Zero WSL-specific handling. No `uname -r` checks, no conditional cron/systemd logic.
When cron isn't running (common WSL default), the error message is wrong.

## 2. Upgrade E2E (Existing Machine)

### Current path

1. `uv tool uninstall aya-ai-assist`
2. `uv tool install --force --from git+https://github.com/shawnoster/aya aya-ai-assist`
3. `aya status` (verify)
4. `aya schedule install` (re-install hooks — **not automatic**)

### What's safe

- **`~/.aya/` is fully preserved** across uninstall/reinstall. Profile, scheduler,
  alerts, activity, packets, config — all untouched.
- **Schema is forward-compatible.** Both profile.json and scheduler.json have
  `schema_version` fields with migration logic and graceful degradation warnings.
- **`aya schedule install` is idempotent.** Detects existing hooks via markers,
  returns `already_present=True` rather than failing.

### Gaps

| Issue | Severity | Detail |
|-------|----------|--------|
| No version staleness warning | MEDIUM | `aya version` only prints current version; no remote check |
| No post-upgrade hook re-install | HIGH | `aya-refresh` skill runs install but doesn't re-run `aya schedule install` |
| Hook format not versioned | MEDIUM | `CANONICAL_HOOKS` has no version field; format changes won't auto-replace old hooks |
| Python 3.14 blocking | MEDIUM | coincurve 21.0.0 has no cp314 wheel; source build fails. Must pin to 3.12. Not documented for users. |

## 3. Skill Architecture Audit

### Surface inventory

| Surface | Location | Count | Purpose |
|---------|----------|-------|---------|
| CLI commands | `src/aya/cli.py` | 41 | Direct terminal usage |
| Claude Code plugin skill | `.claude-plugin/skills/relay/` | 1 (5 verbs) | Relay operations in Claude sessions |
| MCP tools | `src/aya/mcp_server.py` | 8 | Programmatic tool access |
| Claude Code slash commands | `.claude/commands/aya-*.md` | 7 | Setup/maintenance in Claude sessions |

### Overlap matrix

| Capability | CLI | MCP | Plugin Skill | Notes |
|------------|-----|-----|-------------|-------|
| Send packet | `send`, `dispatch` | `aya_send` | `/relay send` | **3 surfaces, naming inconsistent** |
| Receive/poll | `receive` | `aya_receive` | `/relay check` | **3 surfaces, naming inconsistent** |
| Read packet | `read` | — | `/relay read` | MCP gap |
| Reply | `dispatch --in-reply-to` | `aya_ack` | `/relay reply` | Semantic split: ack vs reply |
| Inbox list | `inbox` | `aya_inbox` | — (via read loop) | Skill gap |
| Relay status | — | — | Python fallback | **CLI gap — no `aya relay status`** |
| Schedule: remind | `schedule remind` | `aya_schedule_remind` | — | Skill gap |
| Schedule: watch | `schedule watch` | `aya_schedule_watch` | — | Skill gap |
| Pairing | `pair`, `trust` | — | — | CLI only |
| Config | `config set/show` | — | — | CLI only |
| Packet history | `packets` | — | — | CLI only |

### Naming inconsistencies

| Concept | CLI | MCP | Skill |
|---------|-----|-----|-------|
| Send | `send` (pre-built) / `dispatch` (compose+send) | `aya_send` | `/relay send` |
| Poll | `receive` | `aya_receive` | `/relay check` |
| Reply | `ack` or `dispatch --in-reply-to` | `aya_ack` | `/relay reply` |

### Call patterns

- **Plugin skill** → subprocess CLI (`aya receive`, `aya dispatch`, etc.)
- **Home workspace skills** → subprocess CLI (`aya status`, `aya dispatch`, etc.)
- **MCP tools** → Python library functions (native, no subprocess)

This means CLI signature changes break skills but not MCP. The relay skill
(429 lines) is larger than most CLI commands and hardcodes CLI flags.

### Home workspace integration

5 home skills call aya CLI directly:
- `/session` → `aya status`, `aya context`, `aya inbox`
- `/next` → `aya receive`, `aya status`
- `/status` → `aya status -f json`
- `/pack-for-home` → `aya dispatch`
- `/session-learnings` → references aya architecture docs

| 6 | Document Python 3.14 limitation in README | `README.md` | MEDIUM |

### Architecture decisions (resolved 2026-04-16)

| # | Decision | Detail |
|---|----------|--------|
| A | **Consolidate into `/relay` + `/aya`** | `/relay` stays focused on communication (already done). Remaining slash commands (`setup`, `pair`, `status`, `refresh`, `watch`) consolidate into one `/aya` router skill. Domain-bound grouping. |
| B | **Rename `dispatch` → `send`, old `send` → `send-raw`** | Clean break, no transition period, no hidden alias. The common operation gets the obvious name. Aligns CLI with MCP `aya_send` and skill `/relay send`. |
| C | **Hybrid: skills describe intent, Claude picks surface** | Skills reference MCP tools as preferred, CLI as fallback. Close MCP gaps first (`read`, `pair`, `config`, `packets`, `relay status`) so the preference is meaningful. |
| D | **Add `aya relay status` CLI command** | Closes the CLI gap. Natural fit alongside existing `relay list/add/remove`. MCP can wrap it. Skill stops needing Python fallback. |
